# 我给 Codex 做了一个本地扩展中心

Codex 用久了以后，插件、skill、MCP、本地工具会越装越多。刚开始还记得每个工具是干什么的，过一阵子就容易变成这样：

- 这个 skill 到底叫什么？
- 我装过几个同名版本？
- 这个任务应该用哪个插件？
- 哪些是项目里的，哪些是全局的？
- 第三方插件有没有检查过更新？
- 自己写的 skill 能不能自动出现在一个统一入口里？

所以我做了一个本地网页应用：**Codex Extension Hub（Codex 扩展中心）**。

它不是在线市场，也不是云端推荐系统。它更像你电脑上的 Codex 扩展工作台：只读扫描本机已经存在的扩展元数据，然后整理成一个能搜索、能筛选、能看详情、能做本地推荐的页面。

项目地址：

```text
https://github.com/Daviddwt/codex-extension-hub
```

## 它能做什么

打开以后，你可以看到几类信息：

- 本机一共发现了多少 Plugin、Skill、MCP、App/Connector 和 Hook；
- 每个扩展安装在哪个范围，是全局的、项目级的，还是来自 marketplace；
- 每个扩展的调用名、用途说明、路径、启用状态和提示词线索；
- 哪些扩展有配置提醒，比如同名重复、元数据缺失、marketplace 记录不完整；
- 第三方 marketplace 插件的检查和更新记录；
- 新建的本地 plugin 或 skill 在重新扫描后，也能进入这个工作台；
- 输入“我要做什么”，让它根据本地元数据推荐可能合适的扩展。

它的边界也很明确：不执行插件脚本，不启动 MCP Server，不上传本机扩展数据。推荐框目前是本地元数据打分，不是外部大模型。

## 推荐安装方式：把目标说给 agent

我不建议在公开教程里放“复制这一行脚本直接执行”的安装方式。更适合 agent 时代的做法，是把目标、仓库地址和安全边界说清楚，让本地 agent 自己读取说明、安装依赖、启动页面并完成验证。

你可以直接这样对 agent 说：

```text
请给我的 Codex 安装以下本地工具：
https://github.com/Daviddwt/codex-extension-hub

请先自行读取 README 和安全注意事项，下载或克隆最新版本，安装依赖，
运行本地扫描，启动网页工作台，打开浏览器并完成页面可用性验证。

不要执行扫描到的 Plugin、Skill、Hook 或 MCP 脚本。
不要上传本机扩展数据。
```

如果你想自己逐步操作，GitHub README 里也保留了透明的手动命令。教程正文不放远程安装脚本。

以后写其它插件教程，也可以直接套这个自然语言模板：

```text
给我的 Codex 安装以下地址的插件：
https://github.com/Daviddwt/solution-factory

请自行读取注意事项、下载最新分享包、安装插件、
打开网页工作台并完成验证。

不要执行未知脚本；如果需要权限、登录态或本机路径，请先说明再继续。
```

## 第一次打开会看到什么

首页是一张本地扩展仪表盘。

顶部统计卡片可以帮你快速看清扩展结构：总数、Plugin 数、Skill 数、MCP 数、项目级数量、全局数量、已禁用数量和异常提醒数量。点击这些卡片，下方列表会跟着筛选。

比如点 **异常**，列表会只显示有配置提醒的扩展。这里的“异常”不是说插件坏了，而是提醒你：可能有同名重复、缺少 `SKILL.md`、元数据不完整，或者 marketplace 记录和本地缓存对不上。

下面是搜索和浏览区。你可以按名称、用途、调用名、项目路径或 marketplace 搜索，也可以切换卡片视图、表格视图、按项目、按分类、按插件来源来查看。

## 用任务找插件

我最常用的是推荐框。

比如输入：

```text
我要做一个适合发在公众号、X 和论坛的 AI 使用技巧教程，需要封面、有文字、有配图
```

扩展中心会根据已扫描到的扩展名称、用途说明、分类和提示词线索做本地打分，然后推荐可能适合的插件或 skill。

这一步不会把你的任务描述上传到云端。它只是帮你从本机已有扩展里快速缩小范围。

## 整理重复和配置提醒

如果你经常安装、复制或自己改 skill，很容易出现同名扩展分散在多个目录里的情况。扩展中心会把这类问题标出来。

常见提醒包括：

- 同名扩展存在多个安装位置；
- skill 缺少 `SKILL.md`；
- plugin 元数据不完整；
- marketplace 记录和本地缓存不一致。

这些提醒适合定期看一眼，尤其是在你准备把某个插件或 skill 分享给别人之前。

## 第三方插件更新

扩展中心可以记录第三方 marketplace 插件的更新检查结果。

默认策略是保守的：只处理符合策略的第三方插件，自建插件、`local-plugins`、系统运行时托管插件都会跳过，避免把你本地开发中的东西改掉。

如果你想做自动化，可以让本地 agent 按 README 里的说明配置每周日凌晨 3 点执行检查。页面会展示最近一次检查结果和每个扩展的更新记录。

## 适合放在哪些场景里

这个工具最适合三类人：

- 自己装了很多 Codex 插件和 skill，想知道“到底有什么”的人；
- 团队里有人维护公共 skill，希望大家能看到统一入口的人；
- 经常做教程、方案、PPT、审查、代码协作，想按任务快速找到合适扩展的人。

如果你只是偶尔用一个插件，可能暂时不需要它。但只要扩展数量开始变多，一个本地可见的目录就会省很多时间。

它全程本地运行，不执行插件脚本，不上传数据。安装时也不需要复制远程脚本，直接把 GitHub 地址和安全要求告诉本地 agent，让 agent 自己读取说明、安装、启动并验证。
